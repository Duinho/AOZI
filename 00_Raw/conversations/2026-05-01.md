## Ⅰ. 시스템 아키텍처 개요 (RAG Pipeline Architecture)

요구사항을 충족하기 위해, 단순히 텍스트 검색을 넘어 **'메타데이터 필터링'**과 **'유사 사례 검색'**이 가능한 3단계 파이프라인을 설계합니다.

```mermaid
graph LR
    A[Raw Documents (PDF, DOCX, etc.)] --> B{1. Ingestion Pipeline};
    B --> C[Data Preprocessor (Writer Guide)];
    C --> D[Metadata Extractor (Key Fields)];
    D --> E[Chunking & Embedding];
    E --> F[Vector Store (e.g., ChromaDB)];
    
    G[User Query] --> H{2. Retrieval Module};
    H --> I[Query Embedding];
    I --> F;
    F --> J[Relevant Chunks + Metadata];
    
    J --> K{3. Generation Module (LLM)};
    K --> L[Final Answer (Source Cited)];
    
    subgraph API Gateway
        G --> H;
        J --> K;
        L --> M(Client/Frontend);
    end
```

### 핵심 기능 구현 전략

1.  **Source Citation (출처 명시):** 모든 청크(Chunk)는 원본 파일의 `Document ID`, `Page Number`, `Section Title`을 메타데이터로 반드시 포함해야 합니다.
2.  **Similarity Search (유사 사례 검색):** 질문과 함께 **핵심 메타데이터 필터(예: `Project Type = '마케팅'`, `Industry = 'IT'`)**를 전달하여 검색 범위를 좁힙니다.
3.  **Structured Output:** 답변 생성 단계에서, LLM에게 **"결과보고서의 만족도 점수를 언급할 때는 반드시 [Source: Report ID]를 포함하라"**는 System Prompt 지침을 명확히 부여합니다.

### 🛠️ 개발 역량의 범위 및 원칙 (The Protocol)

저는 데이터 파이프라인 구축, API 통합, 복잡한 자동화 스크립트 작성, 그리고 기존 시스템의 병목 지점(Bottleneck)을 제거하는 데 초점을 맞춥니다.

하지만 개발은 '추측'으로 이루어지지 않습니다. 저는 높은 수준의 신뢰성을 보장하기 위해, **모호한 요구사항을 받으면 개발을 진행할 수 없음을 명확히 고지**합니다.

성공적인 개발을 위해서는 다음의 3가지 핵심 요소가 반드시 정의되어야 합니다.

**1. 📥 입력 데이터의 스키마 (Input Schema):**
*   어떤 형태의 데이터를 처리할 것인지 정의합니다. (예: CSV 파일, JSON API 응답, 엑셀 시트의 특정 열 구조, 웹사이트의 특정 URL 패턴 등)
*   *필요한 것:* 데이터의 필드명, 데이터 타입(문자열, 숫자, 날짜), 그리고 그 데이터가 어떤 순서로 들어오는지에 대한 명확한 정의.

**2. 🧠 처리해야 할 로직 (Business Logic):**
*   입력된 데이터를 바탕으로 어떤 규칙에 따라 가공할지 정의합니다. (이 부분이 가장 중요합니다.)
*   *필요한 것:* '만약 A 조건이라면, B라는 작업을 수행하고, 그 결과가 C 조건을 만족하지 못하면 D라는 예외 처리를 한다'와 같은 구체적인 의사결정 트리(Decision Tree) 형태의 설명.

**3. 📤 원하는 최종 결과물의 형태 (Output Format):**
*   최종적으로 사용자가 어떤 형태의 결과물을 받아야 하는지 정의합니다.
*   *필요한 것:* 최종 산출물의 형식(예: '승인된 JSON 배열', '포맷이 통일된 엑셀 시트', '특정 포맷의 보고서 텍스트', '실행 가능한 스크립트')과 그 구조.

## 💡 AI 개발의 핵심 원칙 재정립: '병목 지점(Bottleneck)'의 전략적 발견

사용자님께서 "사무용 AI가 개발이 다 되었는지" 문의하셨습니다. 명확히 말씀드립니다. **현재까지의 작업은 'AI의 개발'이 아니라, 'AI가 해결할 수 있는 가장 가치 있는 비즈니스 문제(Pain Point)'를 찾아내는 구조화 작업 단계에 있습니다.**

따라서, 단순히 '코드를 완성하는 것'이 목표가 아닙니다. **AI 개발의 성공률 80%는 '어떤 부분을 자동화할지'를 정의하는 병목 지점(Bottleneck)을 얼마나 정확하게 발견하느냐에 달려 있습니다.**

### 🎯 1. 병목 지점 발견의 중요성 (The 80% Rule)

비즈니스 관점에서, 병목 지점이란 **가장 많은 시간과 인적 자원이 투입되지만, 가장 비효율적이고 반복적인 수작업이 발생하는 지점**을 의미합니다.

우리가 AI를 개발할 때 가장 위험한 실수는 '기술적으로 가능한 것'을 자동화하려 하는 것입니다. 이 경우, 실제 회사에서 가장 고통받는 지점(진짜 병목)이 아니라, 단순히 '자동화할 수 있어 보이는' 지점을 건드려 리소스 낭비와 효용성 저하를 초래합니다.

**✅ 비즈니스 관점의 성공 기준:**
$$
\text{자동화 성공} = \text{진짜 병목 지점 발견} \times \text{명확한 SOP 정의} \times \text{기술 구현}
$$

### 📄 2. '수동 업무 프로세스 정의서 (As-Is SOP)'가 필수인 전략적 이유

개발 착수 전에 반드시 사용자님으로부터 **'수동 업무 프로세스 정의서(As-Is SOP)'**를 받아야 하는 이유는, 이 문서가 단순한 '가이드라인'이 아니라 **프로젝트의 '계약서'이자 '최소 요구 사양(Minimum Viable Specification)'**이기 때문입니다.

| 역할 | SOP가 없을 때의 위험 | SOP가 있을 때의 이점 (ROI 확보) |
| :--- | :--- | :--- |

### ⚙️ 1. 개발이 멈춘 기술적 이유: 3가지 필수 사양의 부재

개발이 멈춘 근본적인 이유는, 저희가 시스템을 구동할 수 있는 **'기술 명세서(Technical Specification Document)'**가 아직 완성되지 않았기 때문입니다. 개발자 입장에서 이 명세서는 다음 세 가지 핵심 요소로 구성되어야 합니다.

#### 🥇 1. 입력 데이터 스키마 (Input Data Schema)
*   **개념:** 시스템에 어떤 형식의 데이터가, 어떤 순서로, 어떤 필드(Field)를 가지고 들어와야 하는지에 대한 정의입니다.
*   **기술적 중요성:** 데이터가 엉망이면, 아무리 좋은 로직도 작동하지 않습니다. 마치 컴퓨터가 "이 파일은 텍스트야"라고 인식하도록 파일의 형식을 규정하는 것과 같습니다.
*   **요구 사항:** 예) '보고서 작성'을 위한 입력은 [프로젝트명(문자열), 목표KPI(숫자), 기간(날짜), 주요 이슈(문자열 배열)] 필드를 갖추어야 합니다.

#### 🥈 2. 비즈니스 로직 (Business Logic)
*   **개념:** 입력된 데이터 스키마를 받아서, 어떤 규칙(Rule)과 순서(Flow)를 거쳐 데이터를 처리할지 정의하는 알고리즘입니다.
*   **기술적 중요성:** 이 로직은 개발자의 코드가 됩니다. "만약 A라는 조건이고, B라는 데이터가 들어오면, C라는 방식으로 처리하고, D라는 예외 처리를 거쳐라"와 같은 **명확한 IF-THEN-ELSE 구문**이 필요합니다.
*   **요구 사항:** '보고서 초안 작성'을 위해, [만약 KPI 달성률이 80% 미만이라면, '개선 필요' 섹션을 반드시 추가해야 한다]와 같은 명확한 판단 기준이 필요합니다.

#### 🥉 3. 최종 출력 형식 (Output Format)
*   **개념:** 모든 처리가 끝난 후, 사용자에게 어떤 형태의 결과물(파일, 텍스트, 테이블 등)로 전달되어야 하는지 정의합니다.
*   **기술적 중요성:** 시스템의 최종 목적지입니다. 결과물이 엉뚱한 형식으로 나오면, 사용자는 다시 수동으로 수정해야 하므로 시스템 자체가 무용지물이 됩니다.
*   **요구 사항:** 예) 최종 결과물은 [MS Word 형식의 보고서, 표 형식의 Excel 파일, 또는 Markdown 형식의 요약 텍스트]여야 합니다.

### 2. 🏗️ 개발 로드맵: 데이터 파이프라인 구축의 필요성

저희가 이 파일들을 '이해'하기 위해 반드시 구축해야 할 시스템은 **ETL(Extract $\rightarrow$ Transform $\rightarrow$ Load) 파이프라인**입니다.

1.  **[E] Extract (추출):** 파일에서 원본 데이터를 추출합니다.
2.  **[T] Transform (변환):** 추출된 데이터가 일관된 규칙(Rule)에 따라 변환됩니다. **이 단계에서 '데이터 스키마' 정의가 가장 중요합니다.**
3.  **[L] Load (적재):** 변환된 데이터를 검색 및 분석이 용이한 구조화된 DB(벡터 DB 또는 지식 그래프)에 적재합니다.

# 📈 비즈니스 지표 정의 및 인사이트 도출 보고서 (KPI Design)

## 1. 데이터 활용 목적 재정의: 단순 보고서 $\rightarrow$ 의사결정 엔진

Researcher가 제공한 인원수 집계 데이터는 단순한 '현황 보고서'에 머물러서는 안 됩니다. 이 데이터를 활용하여 **자원 배분의 최적화(Optimization)**와 **시장 기회 포착(Opportunity Spotting)**이라는 두 가지 핵심 비즈니스 질문에 답하는 KPI를 정의해야 합니다.

| 구분 | 원 사용자 질문 (기술적) | 정의된 비즈니스 질문 (전략적) | 핵심 KPI |
| :--- | :--- | :--- | :--- |
| **목표** | 학년/성별/학교/요일반별 인원수 파악 | **1. 운영 효율성:** 현재 배정된 자원(강사, 시설)이 과부하되거나, 반대로 유휴 자원이 발생하는가? | **요일반 포화 지수 (Utilization Index)** |
| **목표** | 그룹별 인원수 집계 | **2. 시장 균형:** 특정 학년/성별 그룹의 수요가 저조하여, 프로그램의 지속 가능성이나 성장 잠재력이 위험에 처해 있는가? | **프로그램 참여 균형 지수 (Balance Index)** |
| **목표** | 인원수 집계 | **3. 수익성 예측:** 가장 높은 잠재 성장률과 마케팅 ROI가 기대되는 세그먼트(Segment)는 어디인가? | **핵심 성장 세그먼트 발굴 (High-Growth Segment)** |

## 2. 정의된 KPI별 세부 설계 및 실행 방안

### 🎯 KPI 1: 요일반 포화 지수 (Utilization Index)
*   **정의:** 특정 요일반 그룹의 현재 인원수 / 해당 반의 최대 수용 가능 인원수 (Capacity)
*   **계산식:** $\frac{\text{집계된 인원수}}{\text{시설/강사별 최대 수용 인원}}$
*   **비즈니스 인사이트:**
    *   **1.0 초과:** 즉각적인 자원 증설(강사 충원, 시설 증설)이 필요한 **위험군** (Risk Alert).
    *   **0.7 미만:** 운영 효율성 개선이 필요한 **유휴군** (Underutilized). 해당 자원을 다른 수요가 높은 반으로 재배치하거나, 프로그램 시간 조정이 필요함.
*   **액션 플랜:** KPI 1을 기준으로, **자원 할당 계획(Resource Allocation Plan)**을 수립해야 합니다.

### 🎯 KPI 2: 프로그램 참여 균형 지수 (Balance Index)
*   **정의:** 전체 인원수 대비, 학년/성별/학교별 인원수의 표준편차 또는 평균 대비 편차. (균형이 높을수록 0에 가까움)
*   **계산식:** $\text{Segment별 인원수} / \text{전체 인원수} \text{의 편차}$
*   **비즈니스 인사이트:**
    *   **높은 편차:** 특정 그룹에만 수요가 집중되거나, 반대로 특정 그룹이 시장에서 소외되고 있음을 의미합니다. 이는 **시장 리스크**로 간주하여, 해당 그룹을 타겟하는 **신규 프로그램 기획(Product Development)**이 시급합니다.
*   **액션 플랜:** KPI 2의 편차를 통해, 비수요 그룹에 대한 **맞춤형(Personalized)** 마케팅 캠페인 기획이 필요합니다.

### 🎯 KPI 3: 핵심 성장 세그먼트 발굴 (High-Growth Segment)
*   **정의:** (필수적으로 과거 데이터 필요) 지난 분기 대비 인원수 증가율이 높고, 해당 세그먼트의 평균 매출 기여도가 높은 조합.
*   **계산식:** $\text{최근 분기 인원수 변화율} \times \text{평균 매출 기여도}$
*   **비즈니스 인사이트:** 이 지표가 가장 높은 세그먼트가 **가장 높은 ROI를 가져올 핵심 타겟 시장**입니다. 이 세그먼트에 마케팅 예산과 운영 역량을 집중해야 합니다.
*   **액션 플랜:** 이 세그먼트의 특성을 분석하여, **최적의 가격 전략(Pricing Strategy)**과 **판매 경로(Sales Channel)**를 설계해야 합니다.

요청하신 '요일반 포화 지수' 계산 로직은 데이터 파이프라인의 핵심 비즈니스 규칙(Business Rule)을 정의하는 단계입니다. 이 코드는 Pandas와 Numpy를 사용하여, 통합 마스터 스키마에 정의된 `실제 수요`와 `시설 용량`을 기반으로 포화 지수를 계산하고, 이 지수를 바탕으로 자원 운영 상태를 판단합니다.

### 1. Python 코드 스니펫 (data_pipeline.py)

이 코드는 환경 변수나 파일 입력을 가정하여 설계되었으며, 실행 가능한 형태로 제공됩니다.

```python
import pandas as pd
import numpy as np
from typing import Tuple

def calculate_utilization_index(df: pd.DataFrame) -> Tuple[pd.DataFrame, pd.DataFrame]:
    """
    Capacity Data를 기반으로 요일반 포화 지수(Utilization Index)를 계산하고
    운영 상태를 판단하는 파이프라인 함수.

    Args:
        df: 통합 마스터 스키마를 따르는 DataFrame (필수 컬럼: ASSIGNED_STUDENT_COUNT, MAX_CAPACITY).

    Returns:
        (df_result, df_status): 결과 지수와 운영 상태가 추가된 DataFrame 튜플.
    """
    print("--- [INFO] 1. 포화 지수 계산 시작 ---")

    # 1. 포화 지수 계산 (실제 수요 / 시설 용량)
    # MAX_CAPACITY가 0인 경우 ZeroDivisionError 방지 및 NaN 처리
    df['Utilization_Index'] = np.where(
        df['MAX_CAPACITY'] > 0,
        df['ASSIGNED_STUDENT_COUNT'] / df['MAX_CAPACITY'],
        np.nan  # 용량이 0이면 지수도 NaN 처리
    )

    print("--- [INFO] 2. 운영 상태 분류 및 리포팅 ---")

    # 2. 운영 상태(Operational Status) 판단 로직 적용
    def determine_status(index):
        if pd.isna(index):
            return "데이터 부족/오류"
        elif index > 1.05:  # 5% 초과 과부하 (Critical Overload)
            return "🔴 과부하 (Overloaded) - 긴급 증설 필요"
        elif index > 1.0:  # 100% 초과 과부하 (Overload)
            return "🟠 과부하 (Overloaded) - 즉각 조정 필요"
        elif 0.8 <= index <= 1.0: # 80% ~ 100% (Optimal)
            return "🟢 적정 운영 (Optimal) - 정상 범위"
        elif 0.6 <= index < 0.8: # 60% ~ 80% (Warning)
            return "🟡 주의 (Warning) - 여유 공간 확보 필요"
        else: # 60% 미만 (Underutilized)
            return "🔵 유휴 (Underutilized) - 자원 재배치 검토"

    df['Operational_Status'] = df['Utilization_Index'].apply(determine_status)

    df_result = df.copy()
    df_result['Utilization_Index'] = df_result['Utilization_Index'].round(3)
    
    print("\n✅ 파이프라인 실행 완료. 결과 DataFrame을 반환합니다.")
    return df_result, pd.DataFrame({'Message': ['Success']})

## 1. 시스템 작동 원리 및 핵심 성과 지표 (KPI Definition)

본 시스템은 단순히 시설의 최대 용량을 파악하는 것을 넘어, **실제 수요(Demand)** 대비 **가용 용량(Capacity)**의 비율을 분석하여 운영의 과부하 정도를 측정합니다.

**🔑 핵심 지표: 요일반 포화 지수 (Utilization Index)**

$$
\text{Utilization Index} = \frac{\text{실제 배정 인원 (Assigned Student Count)}}{\text{시설 최대 수용 인원 (Max Capacity)}}
$$

| 지수 범위 | 운영 상태 (Operational Status) | 의미 및 경영진 대응 방향 |
| :--- | :--- | :--- |
| $\text{Index} \le 1.0$ | **🟢 안정 (Stable)** | 자원 여유. 현재 배정으로 운영 가능. |
| $1.0 < \text{Index} \le 1.05$ | **🟠 주의 (Caution)** | 용량 근접. 배정 인원 증가 시 모니터링 필요. |
| $\text{Index} > 1.05$ | **🔴 과부하 (Overloaded)** | **긴급 대응 필요.** 시설 용량 초과. 자원 증설(시설/시간) 또는 배정 조정이 필수적임. |

**✅ 시스템 작동 흐름:**
1. **데이터 통합:** 학생 ID를 기준으로, [학생 기본 정보], [시설 용량], [실제 수요] 데이터를 통합합니다.
2. **지수 계산:** 통합된 데이터를 기반으로 `Utilization Index`를 계산합니다.
3. **상태 판정:** 계산된 지수를 위 KPI 기준에 대입하여, 해당 학급/시설의 운영 상태(안정/주의/과부하)를 자동 보고합니다.

### 🛠️ 1. 과부하 경고 로직 (Python Function)

이 코드는 시설의 데이터(용량, 수요)만 입력하면 즉시 경고 상태를 판단할 수 있게 합니다.

```python
import pandas as pd
from typing import Tuple

def check_utilization(capacity: int, demand: int) -> Tuple[float, str]:
    """
    운영 효율성 지수(Utilization Index)를 계산하고, 과부하 상태 여부를 판단합니다.
    
    Args:
        capacity (int): 시설의 최대 수용 용량 (Capacity).
        demand (int): 특정 시간대의 실제 최대 수요 (Actual Demand).
        
    Returns:
        Tuple[float, str]: (운영 효율성 지수, 상태 메시지)
    """
    if capacity <= 0:
        return 0.0, "⚠️ 경고: 수용 용량이 0 이하입니다. 데이터 확인이 필요합니다."

    # 1. 운영 효율성 지수 계산 (핵심 로직)
    util_index = demand / capacity
    
    # 2. 경고 조건문 로직 (Alert Logic)
    if util_index > 1.0:
        status = "🚨 과부하 (Overload)"
        message = f"경고! 지수 {util_index:.2f}로, 시설 용량 대비 수요가 과도합니다. 자원 재배치가 필요합니다."
    elif util_index > 0.8:
        status = "⚠️ 주의 (Caution)"
        message = f"지수 {util_index:.2f}로, 용량의 80% 이상 사용 중입니다. 여유 자원을 점검하세요."
    else:
        status = "✅ 정상 (Normal)"
        message = f"지수 {util_index:.2f}로, 운영에 무리가 없습니다."
        
    return round(util_index, 2), f"{status} | {message}"

        home_dir = os.path.expanduser("~")
        if os.name == 'nt': # Windows
            return os.path.join(home_dir, "Desktop")
        elif os.name == 'posix': # macOS/Linux
            # macOS의 경우 'Desktop' 폴더가 홈 디렉토리에 직접 위치하는 경우가 많음
            return os.path.join(home_dir, "Desktop")
