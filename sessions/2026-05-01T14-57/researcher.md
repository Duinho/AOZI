## Ⅰ. 목표 재확인 및 접근 방식

**1. 목표:** 사내 모든 비정형 문서를 학습하여, 질문에 대해 **① 근거 문서 인용, ② 맥락적 유사 사례 제시, ③ 구조화된 지표(KPI, 만족도) 답변**이 가능한 Single Source of Truth (SSOT)를 구축합니다.
**2. 접근 방식:** 단순히 파일을 많이 넣는 것(Data Volume)을 넘어, **'어떤 정보가 어디에 있는지'**를 정의하고 **'그 정보를 어떻게 뽑아낼지'**를 구조화하는 데 초점을 맞춥니다. (Metadata-First Approach)

## Ⅱ. 데이터 소스 인벤토리 및 관리 방안 (Data Source Inventory)

| 데이터 유형 | 예상 저장 위치 (Source) | 파일 형식 (Format) | 접근 권한 (Access) | 수집 난이도 | 비고 (Governance) |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **결과보고서** | 사내 공유 드라이브 (Project/Archive 폴더) | PDF, DOCX, XLSX | 프로젝트 담당자 $\rightarrow$ 전사 공개 (읽기 전용) | 중하 | 가장 핵심 데이터. 메타데이터 추출 필수. |
| **사업 기획/제안서** | 사내 공유 드라이브 (Planning/Proposal 폴더) | PPTX, DOCX | 관련 부서 및 팀장급 이상 | 중 | 목표, 범위, 예산 등 *계획 단계*의 메타데이터 추출. |
| **회의록** | 노션/원드라이브/공유 드라이브 (Meeting Minutes) | DOCX, TXT, 이미지(OCR 필요) | 참여자 전원 | 중상 | '결정 사항(Decision)'과 '다음 액션(Action Item)' 분리 추출이 중요. |
| **만족도/피드백** | 설문 툴 데이터베이스 (Google Forms, SurveyMonkey) | CSV, XLSX | 전사 공개 | 하 | 정량적 지표(Score)와 정성적 피드백을 구조화해야 함. |
| **이메일 기록** | 사내 메일 서버 아카이브 | EML, PST (API 연동 필요) | 전사 공개 (필요 시) | 상 | 특정 프로젝트 관련 핵심 의사소통만 추출하는 선별적 접근 필요. |

## Ⅲ. 문서 유형별 상세 데이터 구조화 방안 (Metadata Schema Definition)

각 문서 유형별로 반드시 추출하고 표준화해야 할 필수 정보(메타데이터) 항목을 정의합니다. 이 항목들이 AI가 검색할 때 필터링 조건이나 답변의 근거가 됩니다.

### 1. 📑 결과보고서 (Result Report) 구조화
| 필드명 (Field Name) | 데이터 타입 | 추출 필요성 | 활용 예시 |
| :--- | :--- | :--- | :--- |
| **프로젝트 고유 ID** | String | 필수 | 모든 기록의 최상위 키(Key) 역할. |
| **사업명** | String | 필수 | 검색 키워드. (예: 24년 상반기 마케팅 캠페인) |
| **프로젝트 기간** | Date Range | 필수 | 기간 필터링 (예: 2023년 1월 ~ 2023년 6월). |
| **핵심 목표 (Goal)** | Text | 필수 | 사업의 존재 이유. (예: 시장 점유율 5% 증가) |
| **KPI 달성률** | Number (%) | 필수 | 정량적 성과 지표. (예: 95%) |
| **주요 성공 요인 (Success Factor)** | List/Text | 필수 | AI가 '왜 성공했는지' 분석하는 근거. (예: A 채널 집중, 팀워크 강화) |
| **주요 실패/개선 요인 (Failure/Lesson)** | List/Text | 필수 | AI가 '무엇을 개선해야 하는지' 제시하는 근거. (예: 초기 시장 조사 부족) |
| **최종 만족도 점수** | Number (1~5점) | 필수 | 정량적 평가. |
| **관련 핵심 인물** | Person Name | 필수 | 담당자, 주요 협력사 등. |
