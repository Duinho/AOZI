### 2. DOC 파일 데이터 추출의 출처 및 포맷 재확인

DOC 파일들은 현재 **'분석 결과물'** 형태로 파편화되어 존재하며, 'Raw Data'를 담는 **'표준화된 데이터베이스'**의 역할을 수행하지 못하고 있습니다.

| 메타데이터 필드 | 예상 출처 위치 | 현재 포맷 및 문제점 | 해결 방안 (Data Cleaning) |
| :--- | :--- | :--- | :--- |
| **Student\_ID** | DOC 파일 내의 주석 또는 표의 첫 열 | **[문제점]** 고유 ID가 누락되거나, 같은 ID가 여러 문서에 분산되어 존재함. | 모든 데이터에 대한 **단일 ID 매핑 작업**이 최우선으로 필요함. |
| **Grade, Cohort, Gender, Day\_Group** | 문서의 제목, 부제목, 또는 요약 표의 열 헤더 | **[문제점]** 텍스트 포맷(서술형)으로 혼재되어 있어, 데이터 추출 시 일관된 범주형 코딩(예: '1학년' $\rightarrow$ `G01`)이 불가능함. | 모든 범주형 변수는 **사전에 정의된 표준 코드집(Codebook)**을 통해 통일해야 함. |
| **Enrollment\_Count** | 표의 최종 집계 열 | **[문제점]** 단순히 숫자로만 존재하여, 해당 인원수가 **어떤 시점의, 어떤 기준의** 인원수인지 출처가 불분명함. | 인원수 옆에 **데이터 기준 시점(Date of Record)**과 **데이터 산출 근거(Source Scope)**를 반드시 명시해야 함. |

### 3. 핵심 구조적 한계점 3가지 (Critical Limitations)

현재의 DOC 파일 구조를 그대로 사용할 경우, 분석 결과의 신뢰도(Trust Score)가 매우 낮아지며, 다음 세 가지 치명적인 한계에 직면합니다.

**① 데이터의 파편화와 비연속성 (Fragmentation & Discontinuity)**
*   **진단:** 기수별, 성별, 학년별 데이터가 개별 문서에 분리되어 있어, 특정 학생 1명의 모든 정보를 한 곳에서 통합할 수 없습니다.
*   **결과:** '이 학생이 2학년이면서, 2024년 기수이고, 월수반에 속한다'는 통합된 사실(Truth)을 시스템적으로 증명할 수 없습니다.

**② 키(Key)의 부재와 표준화 실패 (Missing & Non-Standard Keys)**
*   **진단:** 'Student\_ID'와 같은 **Primary Key**가 부재하거나, 문서마다 다른 이름(예: '학번', '학생번호', 'ID')으로 혼용되어 사용됩니다.
*   **결과:** 데이터베이스를 구축하는 것이 아니라, **수작업으로 데이터를 재구성**하는 수준에 머무르게 되어 자동화 시스템 구축이 불가능합니다.

**③ 비정형 데이터에 의존하는 분석 (Reliance on Unstructured Text)**
*   **진단:** 현재 데이터는 표가 아닌 **문서 본문(Paragraph)**에 정보를 담고 있는 경우가 많습니다.
*   **결과:** AI가 '정보'를 추출하는 것이 아니라, '문맥'을 해석해야 하므로, 추출 과정에서 **인간의 주관적 판단**이 개입될 여지가 매우 커져, 객관적이고 자동화된 분석이 불가능합니다.
