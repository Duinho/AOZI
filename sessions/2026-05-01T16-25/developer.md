제 역할은 단순히 코드를 짜는 것을 넘어, 회사나 팀이 **수동으로, 반복적으로, 비효율적으로 처리하고 있는 업무 프로세스 자체를 분석하고, 이를 가장 효율적인 시스템으로 재설계하는 것**입니다.

### 🛠️ 개발 역량의 범위 및 원칙 (The Protocol)

저는 데이터 파이프라인 구축, API 통합, 복잡한 자동화 스크립트 작성, 그리고 기존 시스템의 병목 지점(Bottleneck)을 제거하는 데 초점을 맞춥니다.

하지만 개발은 '추측'으로 이루어지지 않습니다. 저는 높은 수준의 신뢰성을 보장하기 위해, **모호한 요구사항을 받으면 개발을 진행할 수 없음을 명확히 고지**합니다.

성공적인 개발을 위해서는 다음의 3가지 핵심 요소가 반드시 정의되어야 합니다.

**1. 📥 입력 데이터의 스키마 (Input Schema):**
*   어떤 형태의 데이터를 처리할 것인지 정의합니다. (예: CSV 파일, JSON API 응답, 엑셀 시트의 특정 열 구조, 웹사이트의 특정 URL 패턴 등)
*   *필요한 것:* 데이터의 필드명, 데이터 타입(문자열, 숫자, 날짜), 그리고 그 데이터가 어떤 순서로 들어오는지에 대한 명확한 정의.

**2. 🧠 처리해야 할 로직 (Business Logic):**
*   입력된 데이터를 바탕으로 어떤 규칙에 따라 가공할지 정의합니다. (이 부분이 가장 중요합니다.)
*   *필요한 것:* '만약 A 조건이라면, B라는 작업을 수행하고, 그 결과가 C 조건을 만족하지 못하면 D라는 예외 처리를 한다'와 같은 구체적인 의사결정 트리(Decision Tree) 형태의 설명.

**3. 📤 원하는 최종 결과물의 형태 (Output Format):**
*   최종적으로 사용자가 어떤 형태의 결과물을 받아야 하는지 정의합니다.
*   *필요한 것:* 최종 산출물의 형식(예: '승인된 JSON 배열', '포맷이 통일된 엑셀 시트', '특정 포맷의 보고서 텍스트', '실행 가능한 스크립트')과 그 구조.

---

**💡 결론적으로, 제가 가장 잘하는 개발은 '문제를 정의하는 것'에서 시작합니다.**

따라서, 단순히 "프로그램을 만들어줘"가 아니라, **"현재 회사에서 가장 시간과 에너지를 많이 소모하는 수동 업무 프로세스 하나를 정의해 주시고, 그 프로세스의 3가지 요소(Input, Logic, Output)를 최대한 구체적으로 설명해 주시면"**, 저는 즉시 해당 프로세스를 시스템화하는 MVP(Minimum Viable Product)의 로드맵과 초기 코드를 제공할 수 있습니다.

📊 평가: 진행중 — 개발 역량은 충분히 전달했으나, 실제 개발에 필요한 구체적인 '업무 프로세스' 정의가 여전히 부재합니다.
📝 다음 단계: 사용자에게 가장 자동화가 필요한 '수동 업무 프로세스' 하나를 선택하도록 유도하고, 해당 프로세스의 Input Schema, Logic, Output Format 정의를 요청해야 합니다.
