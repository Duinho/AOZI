## ⚙️ AOZI 파일 이해 가능 여부: 기술적 분석 및 요구사항 정의

사용자님의 질문("이 엑셀들을 이해할 수 있나요?")에 대한 답변은 **"네, 파일의 원본 데이터(Raw Data)는 읽을 수 있습니다. 하지만, AI가 비즈니스 의사결정에 활용할 수 있는 수준으로 '이해'하고 '사용'하려면, 지금부터 말씀드리는 구조화 작업이 필수적으로 선행되어야 합니다."** 입니다.

단순히 텍스트를 읽는 것(Reading)과 그 데이터의 의미와 관계를 파악하여 추론하는 것(Understanding)은 근본적으로 다른 기술적 프로세스입니다.

### 1. 💡 기술적 진단: '읽기'와 '이해'의 차이점

| 구분 | 의미 (사용자 관점) | 기술적 동작 | 한계점 |
| :--- | :--- | :--- | :--- |
| **읽기 (Reading)** | 파일 내용을 확인하는 것 | 파일 시스템에서 바이너리 데이터를 읽어 텍스트로 디코딩 (OCR, CSV 파싱). | 데이터 간의 관계(Relationship) 파악 불가. 데이터의 출처(Source)나 맥락(Context) 정보가 결여됨. |
| **이해 (Understanding)** | 데이터를 활용하여 결론을 도출하는 것 | **데이터 파이프라인(Data Pipeline) 구축** $\rightarrow$ **스키마 검증(Schema Validation)** $\rightarrow$ **지식 그래프(Knowledge Graph) 구축** $\rightarrow$ RAG 검색. | 입력 데이터의 구조가 정의되지 않으면, 데이터의 신뢰도와 일관성 확보가 불가능함. |

**결론:** 현재 파일들은 원본 데이터 셋(Data Set)일 뿐, AI가 학습하고 추론할 수 있는 **구조화된 지식 데이터베이스(Structured Knowledge Base)**가 아닙니다.

### 2. 🏗️ 개발 로드맵: 데이터 파이프라인 구축의 필요성

저희가 이 파일들을 '이해'하기 위해 반드시 구축해야 할 시스템은 **ETL(Extract $\rightarrow$ Transform $\rightarrow$ Load) 파이프라인**입니다.

1.  **[E] Extract (추출):** 파일에서 원본 데이터를 추출합니다.
2.  **[T] Transform (변환):** 추출된 데이터가 일관된 규칙(Rule)에 따라 변환됩니다. **이 단계에서 '데이터 스키마' 정의가 가장 중요합니다.**
3.  **[L] Load (적재):** 변환된 데이터를 검색 및 분석이 용이한 구조화된 DB(벡터 DB 또는 지식 그래프)에 적재합니다.

### 3. 🎯 당장 필요한 정보: Input Schema 정의 요청

**가장 핵심적인 작업은 '스키마(Schema)' 정의입니다.**

스키마란, 테이블(엑셀 시트)의 각 열(Column)이 어떤 **데이터 타입**을 가지며, 어떤 **규칙**에 따라 데이터를 채워야 하는지 정의하는 설계도입니다.

**사용자님께 요청드립니다. AOZI 폴더 내의 엑셀 파일들 중, 가장 핵심이 되는 파일 1~2개를 지정해주시고, 해당 파일의 각 시트(Sheet)별로 아래 형식에 맞춰 스키마를 정의해주십시오.**

**[요청 양식 예시]**

| 시트명 | 컬럼명 (Header) | 데이터 타입 (Data Type) | 필수 여부 (Mandatory?) | 설명 및 제약 조건 (Constraints) |
| :--- | :--- | :--- | :--- | :--- |
| `2024년_실적` | `프로젝트명` | `String` | 필수 | 공백 허용 안됨. (예: A-101) |
| | `담당자` | `String` | 필수 | 사내 직원 명단 기반으로만 입력 가능. |
| | `달성률` | `Float` | 필수 | 0.0 ~ 1.0 사이의 실수만 허용. |
| | `비고` | `String` | 선택 | (자유 서술형) |

이 스키마 정의가 완료되어야만, 저희가 해당 파일들을 파이프라인에 연결하고, 일관성 있는 지식 구조를 구축할 수 있습니다.
