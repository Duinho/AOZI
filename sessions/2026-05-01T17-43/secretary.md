## 📋 에러 로깅 및 보고 절차 정의 (Error Logging & Reporting Procedure)

사용자님의 지적은 시스템의 신뢰성과 투명성 확보 측면에서 매우 중요합니다. 단순한 '실패' 통보만으로는 업무 프로세스 개선이 불가능합니다.

이에 따라, 향후 모든 에이전트 작업 및 시스템 실행 결과에 대해 **'에러 발생 시 원인 추적 및 보고 체계'**를 최우선 과제로 정의하고, 아래와 같이 표준화된 로깅 및 보고 절차를 수립했습니다.

---

### 💡 목표: 실패 원인에 대한 투명성 확보 (Root Cause Transparency)

**목적:** 단순히 '작업 실패'를 알리는 것을 넘어, **'무엇이, 왜, 어떻게'** 실패했는지 구조화된 데이터로 기록하고, 다음 스텝의 개선 방향을 도출하는 것이 목표입니다.

### 🛠️ 표준 에러 로깅 포맷 (Standard Error Log Format)

향후 에이전트가 산출물을 내거나 시스템을 실행할 때, 아래 6가지 항목을 필수로 기록하고 보고해야 합니다.

| 필드명 | 정의 | 필수 입력 데이터 (예시) | 중요도 |
| :--- | :--- | :--- | :--- |
| **1. 실행 시점 (Timestamp)** | 오류가 발생한 정확한 시간 기록. | `2026-05-XXT10:30:00` | ⭐ |
| **2. 요청/입력 데이터 (Input)** | 에러가 발생한 작업을 수행하기 위해 입력된 원본 데이터. | *[스키마 포함]* (예: 엑셀 파일 ID, 사용자 쿼리 전문) | ⭐ |
| **3. 예상 결과 (Expected Output)** | 이 작업을 통해 궁극적으로 얻고자 했던 결과 또는 KPI. | *[KPI 정의]* (예: 'Utilization Index 계산 결과', 'A 그룹의 평균 만족도') | ⭐ |
| **4. 실제 결과/오류 메시지 (Actual Failure)** | 시스템이 반환한 실제 메시지, 혹은 오류 코드. | `ERROR_CODE_400: Missing required parameter 'Date Range'` | ⭐ |
| **5. 근본 원인 분석 (Root Cause Analysis)** | **가장 중요.** 4번 오류 메시지가 발생한 구조적/논리적 이유. | *[분석 예시]* (예: "입력 데이터의 날짜 형식이 통일되지 않아 로직 적용 불가") | ⭐⭐⭐ |
| **6. 권장 조치 및 담당자 (Next Action)** | 이 오류를 해결하기 위해 사용자 또는 개발자가 취해야 할 구체적인 행동. | *[Action]* (예: "데이터 전처리 단계에서 날짜 포맷 통일 작업 필요. 담당: Developer") | ⭐⭐⭐ |

### 🚀 보고 프로세스 (Reporting Flow)

1. **실행:** 에이전트가 작업을 수행합니다.
2. **검증:** 작업 결과가 '예상 결과'와 다를 경우, 에러 발생을 감지합니다.
3. **로깅:** '표준 에러 로깅 포맷'에 맞춰 6가지 항목을 즉시 작성합니다.
4. **보고:** 사용자에게 단순 실패 통보 대신, **'에러 발생 보고서'** 형식으로 해당 로깅 결과를 제시합니다.
