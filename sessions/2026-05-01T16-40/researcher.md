# 📂 데이터 접근 가능성 및 스키마 정의 보고서 (학생 인원수 산출용)

## 1. 데이터 소스 식별 및 분류

요청된 정보(학년, 성별, 학교, 요일반별 인원수)를 산출하기 위해서는 단일 파일만으로는 부족하며, 최소 3가지 유형의 원본 데이터 파일이 필요합니다.

| 파일 유형 | 예상 파일명 (가정) | 포함 정보 | 역할 | 비고 (필수성) |
| :--- | :--- | :--- | :--- | :--- |
| **A. 기본 명단/개인 정보** | `학생_등록_마스터.xlsx` | 학생 ID, 이름, 성별, 학년 (1~3) | 분석의 주체(개인)와 기본 속성(성별, 학년)을 제공. | **필수** |
| **B. 학교/반 배정 정보** | `반편성_배정현황.csv` | 학생 ID, 학교명, 요일반 구분 (월수/화목) | 학생이 속한 그룹(학교, 요일반)을 정의. | **필수** |
| **C. (선택적) 메타 데이터** | `학년_학교_매핑.xlsx` | 학년 코드, 학교명 코드 매핑 테이블 | 데이터 간의 일관성을 높이고 JOIN의 안정성을 확보. | **강력 권장** |

***
**⚠️ 데이터 접근 위험 요소:**
현재 상태로 볼 때, `학생_등록_마스터.xlsx`와 `반편성_배정현황.csv` 간의 **'학생 ID'**를 키(Key)로 활용한 데이터 병합(JOIN) 작업이 필수적입니다. 만약 두 파일의 ID 체계가 다르다면, 데이터 전처리(Cleaning) 단계가 가장 중요합니다.

## 2. 요구 데이터 스키마 정의 (Target Schema)

최종적으로 산출되어야 할 결과물(KPI)을 기준으로, 필요한 통합 데이터셋의 스키마를 다음과 같이 정의합니다.

| 필드명 (Column Name) | 데이터 타입 (Data Type) | 설명 (Description) | 예시 데이터 | 출처 파일 |
| :--- | :--- | :--- | :--- | :--- |
| **`학생_ID`** | VARCHAR / INT | 고유 학생 식별자 (JOIN Key) | S2024001 | A, B |
| **`학년`** | INT | 학년 정보 (1, 2, 3) | 2 | A |
| **`성별`** | VARCHAR | 성별 (남/여) | 여 | A |
| **`학교명`** | VARCHAR | 소속 학교명 | A학교 | B |
| **`요일반_구분`** | VARCHAR | 요일반 배정 그룹 | 월수반 | B |
| **`인원수`** | INT | 최종 집계된 인원수 (COUNT) | 50 | (집계 결과) |

## 3. 데이터 일관성 및 누락 메타데이터 점검

| 항목 | 현재 상태 (가정) | 문제점/위험 요소 | 개선 방안 (Action Item) |
| :--- | :--- | :--- | :--- |
| **데이터 일관성** | 성별/학년 표기 방식이 통일되지 않았을 가능성 존재. | '남성'과 '남', '1학년'과 '1' 등 포맷 불일치로 인한 집계 오류 발생 가능. | **표준화:** 모든 텍스트 필드는 통일된 포맷(예: `성별` = '남', '여' / `학년` = '1', '2', '3')으로 전처리해야 함. |
| **메타데이터 누락** | 학교/요일반 배정이 학년과 직접 연결되어 있지 않음. | 특정 학년의 학생이 여러 학교나 요일반에 분산되어 있을 경우, **최종 배정 우선순위** 기준이 필요함. | **규칙 정의:** '최종 배정 데이터가 최우선'이라는 명확한 비즈니스 규칙을 정의하고, 이를 필터링 조건으로 사용해야 함. |
| **스키마 정의** | `인원수`가 학생 개별 데이터가 아닌, **집계 값**임. | 데이터셋 자체가 개별 레코드(Raw Data)가 아닌, 집계 테이블 형태로 최종 출력되어야 함. | **산출물 정의:** 최종 결과물은 피벗 테이블(Pivot Table) 형태로, 행(Row)은 [학교명, 학년, 성별, 요일반], 열(Column)은 [인원수]가 되도록 목표 설정. |
